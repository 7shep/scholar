## Overview

This change moves the app from source-only builds to a publishable Electron desktop pipeline.

The implementation uses:

- `electron-builder` for packaging and GitHub Release publishing
- `electron-updater` for in-app update checks
- GitHub Actions for OS-specific builds

## Packaging Strategy

### Windows

- Target: `nsis`
- Output: installer `.exe`
- Auto-update metadata: generated by `electron-builder`

### macOS

- Targets: `dmg` and `zip`
- `dmg` is the user-facing download
- `zip` is required so updater metadata can support macOS auto-update

## Auto-Update Strategy

The Electron main process should:

1. Only initialize update checks for packaged builds.
2. Ask `electron-updater` to check GitHub Releases for newer versions.
3. Prompt the user when an update has been downloaded and offer restart-now vs later.

This keeps the behavior simple and avoids development-mode noise.

## GitHub Release Publishing

`electron-builder` should publish to:

- provider: `github`
- owner: `7shep`
- repo: `school-organizer`
- release type: draft

Using draft releases lets CI upload assets first, then the release can be reviewed before being published.

## GitHub Actions

Use separate jobs:

- `windows-latest` builds Windows artifacts
- `macos-latest` builds macOS artifacts

Each job should:

1. check out the repo
2. set up Node
3. run `npm ci`
4. run the release script for its platform

Both jobs publish into the same tag-backed GitHub Release using `GH_TOKEN`.

## Secrets and Signing

### Required for publishing

- default `GITHUB_TOKEN` with `contents: write`

### Windows signing

- optional for initial internal testing, recommended for public release
- workflow should map Windows signing secrets into the `CSC_LINK` and `CSC_KEY_PASSWORD` env vars for the Windows job

### macOS signing/notarization

- required for polished public distribution and reliable macOS auto-update
- workflow should pass:
  - `CSC_LINK`
  - `CSC_KEY_PASSWORD`
  - `APPLE_ID`
  - `APPLE_APP_SPECIFIC_PASSWORD`
  - `APPLE_TEAM_ID`

## Risks

- Unsigned or unnotarized macOS builds will produce a poor install/update experience.
- If the GitHub repo visibility or release permissions change, updater publishing can fail.
- If release assets are built from one OS only, the missing platform will not receive installable downloads.
